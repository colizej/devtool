## 3. Критические пробелы

Раздел 15 описывает **что** делает AI, но не отвечает:

**Рекомендация:** Добавить отдельный раздел AI Engine с: выбором провайдера, примерами промптов, стратегией вызова (on-demand vs scheduled).

---

### 3.2 OAuth токены — нет схемы хранения ❌

Раздел 19 говорит "хранить токены безопасно", но не описывает как:
- `.env` файл? `python-decouple`? `django-environ`?
- Где хранятся `refresh_token` для OAuth2?
- Как обновлять токены при истечении?

**Рекомендация:** Добавить схему: `credentials.json → зашифрованное хранилище в БД или .env + google-auth-oauthlib`.

---

### 3.3 Scheduler — выбор не обоснован для MVP ❌

> "Для MVP достаточно cron"

Cron — проблема:
- Нет retry при падении
- Нет видимости выполнения задач
- Нет логики зависимостей задач

**Рекомендация:** Даже для MVP использовать **django-apscheduler** (без Redis) или **Celery + Redis** (запущен в Docker). Это закладывает правильную архитектуру.

---

### 3.4 Анализ репозитория — размытый раздел ❌

Раздел 14 ("Анализ репозитория") описывает проверки которые **дублируют crawler**:
- meta, robots.txt, sitemap — всё это crawler уже делает
- Что именно читается из Git? Коммиты? History? Файлы?

**Рекомендация:** Либо сфокусировать раздел (например: "парсим HTML-файлы из репозитория для статического анализа"), либо убрать в Phase 2 и не путать с crawler.

## 5. Что подправить

### 5.1 Про HTMX (Раздел 5)

> "HTMX может конфликтовать с Django CSRF"

Это **неверное утверждение**. Django и HTMX интегрируются штатно через `hx-headers` + `{% csrf_token %}`.
Аргумент для отказа от HTMX лучше: "Для сложной JS-логики (Chart.js, динамические фильтры) чистый fetch API даёт больше контроля". Это правда.

### 5.2 SQLite для MVP

Указано "SQLite или PostgreSQL для MVP".
**Рекомендация:** сразу PostgreSQL даже в MVP. SQLite имеет проблемы с concurrency (Celery + Django параллельно). Docker postgres-контейнер решает это за 5 минут.

### 5.3 Модель `SearchConsoleMetrics`

Нет поля `query` (ключевой запрос) и `page` (URL страницы).
GSC API возвращает данные в разрезе query+page — без этих полей не получится анализировать запросы.

```python
# Нужно добавить:
class SearchConsoleMetrics(models.Model):
    project = models.ForeignKey(Project, on_delete=models.CASCADE)
    date = models.DateField()
    query = models.TextField(null=True)   # ← добавить
    page = models.URLField(null=True)     # ← добавить
    clicks = models.IntegerField()
    impressions = models.IntegerField()
    ctr = models.FloatField()
    position = models.FloatField()

    class Meta:
        indexes = [
            models.Index(fields=['project', 'date']),
            models.Index(fields=['project', 'page']),
        ]
```

## 6. Технические риски (дополнение к разделу 22)

| Риск | Вероятность | Митигация |
|---|---|---|
| Google OAuth refresh token истёк | Высокая | Реализовать автообновление токена через `google-auth-oauthlib` |
| GSC API quota (2000 req/day) | Средняя | Кэш в БД, запрос раз в сутки |
| Crawler блокировка (403/429) | Средняя | User-agent, задержки, robots.txt respect |
| AI галлюцинации (ложные рекомендации) | Высокая | Указано в ТЗ — только рекомендации, без авто-изменений ✅ |
| Большие объёмы данных GSC (100K строк/сайт) | Средняя | Пагинация при импорте, архивирование старых данных |
| Sitemap Index (вложенные sitemaps) | Средняя | Рекурсивный парсер + `python-sitemap` |

## 7. Итоговое мнение

Проект **реализуем** — и именно в том стеке, который выбран.

Django отлично подходит для такого dashboard: встроенная админка, ORM, легкая интеграция с Google API.

**Главный совет:** не пытаться реализовать всё сразу.
MVP — это работающий Search Console + одна страница с данными.
Всё остальное приходит итеративно.

> Самая частая ошибка при таких проектах: зависнуть на архитектуре и не запустить ни одной рабочей фичи за месяц.
